O diagnóstico

Por que IA "padrão" não vira operador

Três limitações impedem um Claude genérico de virar infraestrutura de empresa. Entender isso é o primeiro passo.

M

Sem Memória

Claude esquece toda conversa por padrão. Sem persistência, você ensina o mesmo contexto toda sessão — como ter um assistente com amnésia diária.


Solucao: MEMORY.md + memory/
C

Sem Consistência

Mesma pergunta, resposta diferente. Para virar processo, cada passo precisa ser versionado, testado e repetível — não depender do humor do modelo.


Solucao: Skills em .claude/skills/
A

Sem Acesso

Terminal exclui não-técnicos. Se só uma pessoa sabe rodar, a IA virou ferramenta pessoal — não infraestrutura de empresa.


Solucao: Dashboard + skills com /comando

Como começar

Os 4 passos para montar seu assessor

Siga nessa ordem. Pular etapas é o erro mais comum. Cada passo tem um "teste de aprovação" — só avance quando o teste passar.

1

Camada 1 · Obrigatória

Context — o assessor conhece o negócio

Teste de aprovacao: abra uma sessão nova (sem histórico) e pergunte "quem é o cliente X, qual fase do projeto, qual o tom de comunicação". Se responder corretamente sem buscar nada, passou.

Configure o CLAUDE.md na raiz do seu projeto e os arquivos de memória. É o "cérebro" que carrega em toda sessão.

Criar: CLAUDE.md
Papel, projetos ativos, regras de comunicação, o que nunca fazer sem perguntar.
Criar: memory/sobre-mim.md
Seu perfil, prioridades, estilo de trabalho, decisões que você pode tomar sozinha.
Criar: stakeholders.yaml
Quem é quem: papel, canal preferido, tom de comunicação, o que precisa saber.
Criar: state.yaml
Fase atual, próxima ação, deliverables, blockers. Atualiza a cada sessão.

Copie os templates da pasta templates/ e preencha com seu contexto. Leva 30-60 minutos na primeira vez.

2

Camada 2 · Paralela ao Passo 3

Connections — alcança os seus sistemas

Teste de aprovacao: peça "me mostra as tasks que vencem essa semana no ClickUp". Se devolver dados reais sem você copiar nada, passou.

Configure as integrações (MCPs) com as ferramentas que você já usa. ClickUp, Gmail e Google Drive cobrem 80% das necessidades.

# .mcp.json — adicione na raiz do projeto
{
  "mcpServers": {
    "clickup": {
      "command": "npx",
      "args": ["-y", "@composio-core/mcp@latest",
               "--toolkits", "clickup"],
      "env": { "COMPOSIO_API_KEY": "SUA_CHAVE" }
    }
  }
}

Opções disponíveis: ClickUp, Notion, Linear, Gmail, Google Drive, Google Calendar, Slack, LinkedIn.

3

Camada 3 · Paralela ao Passo 2

Capabilities — executa workflows com uma frase

Teste de aprovacao: digite /status. Se devolver fase, próximos passos e blockers do projeto sem você escrever nada mais, passou.

Crie skills — arquivos markdown que viram comandos. Uma skill bem feita faz em segundos o que levaria 20 minutos de digitação.

As 3 skills para começar (já incluídas nos templates):

/status

Lê state.yaml e entrega fase, próxima ação, deliverables e blockers em 15 linhas.

/email

Gera rascunho de e-mail no tom certo para o stakeholder indicado. Nunca envia — só rascunha.

/ata

Transforma notas bagunçadas de reunião em ata estruturada + decisões + tasks.

Estrutura de pasta: .claude/skills/status/SKILL.md — é isso. Markdown puro.

4

Camada 4 · Somente depois dos passos 1-3 funcionando

Cadence — roda sozinho, com laptop fechado

Teste de aprovacao: na segunda-feira, antes de você abrir o computador, o assessor já preparou o briefing do dia com decisões, riscos e follow-ups. Sem você pedir.

Configure rotinas automáticas com o comando /schedule no Claude Code. Exemplos práticos:

Briefing matinal (8h): lê ClickUp + Gmail + Calendar, entrega resumo de 5 bullets por e-mail.

Status semanal (sexta 16h): gera relatório por projeto, agrupado por responsável, envia draft para revisão.

Alerta de prazo (2 dias antes): detecta deliverable próximo do vencimento, notifica responsável.
Regra de ouro (Nate Herk): nunca automatize um workflow manual que ainda está quebrado. Primeiro faz funcionar nos passos 1-3, depois automatiza.

Estrutura base

O que criar no seu projeto

Esta é a estrutura mínima para sair do zero com um assessor funcional. Copie a pasta templates/ deste repositório e adapte.

meu-projeto/ ├── CLAUDE.md ← o cérebro: quem você é, seu contexto, regras ├── state.yaml ← fase atual, deliverables, próxima ação, blockers ├── stakeholders.yaml ← quem é quem, canal preferido, tom de comunicação ├── voice.md ← tom por plataforma, frases proibidas, exemplos ├── memory/ │ ├── MEMORY.md ← índice (carregado em toda sessão) │ ├── sobre-mim.md ← seu perfil e preferências │ └── feedback.md ← aprendizados acumulados (atualiza sempre) └── .claude/ └── skills/ ├── status/ │ └── SKILL.md ← /status: estado do projeto em 15 linhas ├── email/ │ └── SKILL.md ← /email: rascunho no tom certo └── ata/ └── SKILL.md ← /ata: notas → ata + decisões + tasks

CLAUDE.md

O arquivo mais importante. Define o contexto que o Claude carrega em toda sessão. Mantenha abaixo de 200 linhas — arquivo longo não é contexto, é ruído.

state.yaml

Foto do projeto agora: fase, próxima ação, deliverables em andamento, blockers. Atualiza a cada sessão com o avanço real.

stakeholders.yaml

Quem é quem: papel, e-mail, canal preferido, tom de comunicação, o que precisa saber e o que não incomodar. Usado pelas skills de comunicação.

voice.md

Tom de comunicação por plataforma (LinkedIn, Instagram, e-mail, WhatsApp). Frases proibidas. Exemplos reais dos seus melhores textos.

MEMORY.md

Índice curto que aponta para os arquivos de memória. Carregado em toda sessão. Máximo 100 linhas. Nunca coloque conteúdo aqui — só ponteiros.

feedback.md

O que funcionou e o que não funcionou. Atualiza a cada vez que algo surpreende — pra bem ou pra mal. É o que faz o assessor melhorar com o tempo.

O que aprendemos na prática

Regras que valem pra todo mundo

Coletadas de Chase H, Nate Herk, Garry Tan e Simon Willison — e confirmadas no uso diário da NTICS.

H

Human-in-the-loop sempre antes de publicar

Nada vai para cliente, rede social ou e-mail sem aprovação humana. O assessor escreve rascunhos; pessoas validam. Os três frameworks concordam nisso sem exceção.

V

CLAUDE.md menor que 200 linhas

Arquivo enorme que ninguém lê não é contexto, é ruído. Estudo com 328 projetos reais mostrou que os que funcionam melhor são curtos, específicos e apontam para docs externas.

B

Boring is beautiful. Workflows beat agents.

Processo determinístico antes de agente autônomo. Uma skill que sempre faz a mesma coisa certa é mais valiosa que um agente criativo que às vezes erra.

200

Não confie no 200 OK

Após criar uma task no ClickUp, confirme que existe. Após criar draft no Gmail, confirme que aparece. O código de resposta de sucesso não garante que a ação foi executada corretamente.

M

Modelo certo para cada tarefa

Haiku para coleta e triagem (25x mais barato). Sonnet para criação e execução. Opus para planejamento estratégico e decisões complexas. Usar Opus pra tudo é como pegar helicóptero pra ir ao mercado.

V

Voice-lock por cliente e plataforma

LinkedIn tem tom diferente do WhatsApp. O projeto Samarco tem voz diferente do projeto Vale. Documente em voice.md — sem isso, o assessor soa genérico em tudo.

O que nao fazer

Anti-patterns comuns

Pular etapas do Four Cs

Criar agentes autônomos (Cadence) antes de ter Context e Capabilities funcionando. Resultado: automação que produz output irrelevante porque não conhece o negócio.

Publicar sem revisão humana

Nenhum dos três frameworks recomenda publicação autônoma para conteúdo de marca. O risco de dano reputacional supera qualquer ganho de velocidade.

CLAUDE.md enorme

Arquivo de 2.000 linhas que tenta cobrir tudo. Na prática, o modelo ignora partes. Melhor: arquivo curto + apontar para docs específicas quando necessário.

Tom igual em todas as plataformas

Chase H: "Se seu LinkedIn e seu X são intercambiáveis, um deles está errado." Cada canal tem gramática própria — force o assessor a respeitar isso.

Creds de produção sem aprovação

78% das empresas dão ao agente acesso total sem revisar ações críticas (survey Gravitee 2026). Sempre use o padrão: agente escreve, humano aprova, sistema executa.

Usar Opus para tudo

Modelo mais caro para coleta de dados é desperdício. Use Haiku para triagem/coleta, Sonnet para criação, Opus só quando o raciocínio complexo justifica o custo.
